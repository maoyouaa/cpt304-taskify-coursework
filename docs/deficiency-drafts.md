## Section 2. Deficiency 1: Accessibility and keyboard usability

### 2.1 Detection
The original Taskify prototype contained several accessibility weaknesses. In the original `signup.ejs`, input fields relied mainly on placeholders rather than robust visible labels, which can reduce compatibility with assistive technologies and make form purpose less clear after typing begins. In addition, the project used multiple EJS partials that contained nested `html`, `head`, and `body` tags, resulting in weak semantic structure across pages. The original dashboard was also a static mockup with inconsistent interactive semantics. These issues were identified through manual inspection of the source structure and should be verified in the final report with `axe DevTools` and a Lighthouse Accessibility audit screenshot from the original version. The defect matters because task-management software is interaction-heavy; if the form structure, focus flow, and labeling are unclear, keyboard-only and screen-reader users will struggle to use core workflows.

### 2.2 Literature
The fix was guided by W3C's WCAG 2.2 understanding materials, especially the principles around perceivable and operable interfaces. WCAG emphasizes that form controls need clear programmatic identification and that interactive content should remain understandable through keyboard navigation and assistive technologies. This literature gave the implementation a concrete direction: replace placeholder-only patterns with explicit labels, keep headings meaningful, preserve semantic grouping, and ensure visible focus treatment instead of treating accessibility as a cosmetic issue. In other words, the literature did not simply suggest "improving usability"; it defined structural expectations that could be translated into code and re-tested with automated tools.

### 2.3 Implementation
The revised version rebuilt the page structure into valid top-level templates with shared partials and introduced explicit `label` elements across the signup flow, search controls, and task-composer form. A visible `:focus-visible` style was also added in `static/styles/main.css`, giving keyboard users a clear indication of which control is active. The dashboard columns now use semantic headings, and the main task actions are represented as real buttons rather than decorative elements. This implementation followed the literature directly: accessible labels improve name computation for form fields, semantic structure improves screen-reader interpretation, and visible focus indicators improve keyboard operability. For the report, the old snippet should show placeholder-driven fields from the original signup page, while the new snippet should show the labeled task form or onboarding form. After the final Lighthouse and axe re-test, include the updated score and the audit screenshot to demonstrate measurable improvement.

## Section 3. Deficiency 2: Missing state persistence

### 3.1 Detection
The original dashboard did not function as a true task-management system. Instead of storing and restoring task data, it rendered hard-coded example cards directly in EJS. This meant that the application provided no actual persistence mechanism and could not preserve user state after refresh. The problem was identified through manual inspection of `views/partials/dashboard/dashboard-cards.ejs`, where task cards were static markup rather than data-driven records, and by observing that the original system had no storage logic in either front-end scripts or server-side handlers. In a productivity application, this is a significant defect because a user expects planning data to remain available between interactions. Without persistence, the system behaves more like a visual mockup than a usable tool.

### 3.2 Literature
The implementation was guided by MDN documentation on the Web Storage API. MDN explains that `localStorage` provides a straightforward client-side persistence mechanism for small amounts of structured interface data that should survive page reloads within the same browser. This literature was appropriate because the coursework version does not require multi-user synchronization or a full hosted database; instead, it needs a reliable, understandable way to preserve state locally and demonstrate engineering reasoning. The source also supports careful storage access, including serializing data, reading it safely, and recovering from malformed stored content.

### 3.3 Implementation
To address this deficiency, a new storage module was created in `static/js/modules/task-store.mjs`. This module defines task normalization, validation, grouping, filtering, metrics, safe loading, and persistence operations. The dashboard script now writes task updates into `localStorage`, restores them on load, and falls back to default seed tasks if no saved data exists or the saved payload is malformed. This is directly connected to the literature: the report can explain that Web Storage was selected because it is lightweight, browser-native, and suitable for a coursework deployment where tasks are intentionally device-local. The "before" snippet should show static task markup from the old dashboard partial, while the "after" snippet should show `safeLoadTasks` or `persistTasks`. In the evidence section, include a refresh demonstration showing that a newly created task remains visible after reloading the page.

## Section 4. Deficiency 3: Weak user feedback and unclear system status

### 4.1 Detection
The original Taskify interface provided almost no meaningful workflow feedback. Because the dashboard was only a static layout, users could not actually create, update, delete, or move tasks, and therefore received no confirmation about whether an action had succeeded, failed, or changed the system state. Even if the layout looked polished, it lacked one of the most basic qualities of interactive software: visibility of what the system is doing. This defect was identified through manual walkthrough of the original pages and source inspection, showing the absence of any feedback state management, action-result messages, or changing metrics.

### 4.2 Literature
This improvement was informed by Nielsen Norman Group's usability heuristics, especially the principle of visibility of system status. The heuristic states that users should always be informed about what is happening through appropriate feedback within a reasonable time. In task-management software, that principle is especially important because users frequently perform small actions whose success must be obvious: saving a task, changing status, clearing a form, or recovering saved work. The literature therefore provided a direct rationale for implementing concise feedback rather than assuming the interface alone was self-explanatory.

### 4.3 Implementation
The revised dashboard now provides visible status banners and outcome messages for task save, update, delete, reset, local-storage recovery, persistence failure, and task-status transitions. It also updates summary metrics for total tasks, tasks in progress, and completed tasks so that board state changes are immediately visible after interaction. These additions are implemented mainly in `static/js/dashboard.js` and rendered within the dashboard template. The implementation follows the literature in a practical way: rather than merely adding decorative alerts, it ensures that each meaningful workflow event produces a timely and relevant response in the UI. In the report, the old state can be described as "no actionable feedback because no real task workflow existed," while the new state should include a snippet of `showMessage(...)` or the form-message/status-banner structure. A screenshot showing the success message after saving a task will work well as evidence.

## Section 5. Deficiency 4: Weak input validation and unsafe content handling

### 5.1 Detection
The original prototype did not provide a real task input pipeline, so there was no robust validation strategy for task content. Once the project was converted into a working dashboard, it became clear that accepting arbitrary raw input without normalization, required-field checks, controlled enums, or duplicate detection would create both quality and safety problems. Empty or whitespace-only titles, inconsistent status values, and uncontrolled content lengths are all examples of input defects that reduce reliability. This was identified by inspecting the absence of task-processing logic in the original project and by testing invalid values during the upgraded implementation phase.

### 5.2 Literature
This fix was guided by the OWASP Input Validation Cheat Sheet. OWASP recommends validating input as early as possible, preferring allowlists and known-good formats instead of loosely accepting anything and trying to clean it later. The source also supports using bounded lengths, normalized values, and constrained categories where appropriate. For this coursework project, that guidance translates well into practical frontend engineering: titles should be mandatory, priority and status should come from approved values, and rendered task content should remain plain text rather than executable HTML.

### 5.3 Implementation
The updated project now validates and normalizes task input inside `static/js/modules/task-store.mjs`. The new logic trims repeated whitespace, requires a non-empty title, limits field lengths, restricts status and priority to valid allowlisted values, validates the date format, and checks for duplicate tasks. The dashboard then renders user content through text nodes rather than inserting HTML into the page, which avoids creating an unsafe rendering path. This implementation follows OWASP's guidance directly: the system no longer trusts raw input blindly, and instead transforms it into a predictable structure before storage and rendering. In the report, the old side can be presented as the absence of validation logic in the original task flow, while the new side should show the `validateTaskInput(...)` or `createTask(...)` function. A good figure here would demonstrate rejection of an empty title or show how duplicate tasks are prevented.
